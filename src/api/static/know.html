💬 Вопрос
№6. Structed errors on frontend
После деплоя streaming upload + очереди при попытке загрузить файл — ошибка "UploadError is not defined" и пустой ответ сервера.
✅ Решение
Проблема: class declarations в JS не hoist'ятся. UploadError был определён после uploadFileXHR, которая его использует. Исправлено: класс перенесён до функции. Также обновлён know.md. Коммит: 924d53a, 104a4bc.
💬 Вопрос
№5. Occular-ocr не работает, падает на Tesseract
После деплоя при загрузке PDF вместо Occular-ocr запускался Tesseract. В логах: OCRPipeline.__init__() got an unexpected keyword argument 'max_workers'. Tesseract обрабатывал одну страницу >2 минут, worker блокировался.
✅ Решение
Новая версия ocr_skel убрала параметр max_workers из конструктора. Убран вызов OCRPipeline(onnx=True, gpu=False, max_workers=workers) → OCRPipeline(onnx=True, gpu=False). Fallback цепочка: Docling + Occular → Docling only → Tesseract. Файл: src/indexing/hybrid_parser.py. Коммит: d2851f8.
💬 Вопрос
№4. Streaming upload + async Queue + ProcessPoolExecutor
Задача: переделать загрузку файлов — убрать await file.read() (весь файл в память), заменить Semaphore(1) на очередь с пулом воркеров, добавить upload_id, cleanup temp-файлов, лимит 500MB.
✅ Решение
Три файла изменено:

1. src/api/routes/upload.py — полная переработка:
• Стриминг в /tmp/uploads/{upload_id}_{filename} по 64KB
• Замена Semaphore(1) на asyncio.Queue + пул воркеров
• _MAX_WORKERS = int(env.MAX_WORKERS) or min(4, cpu_count())
• ProcessPoolExecutor для CPU-bound (OCR, pdf2image)
• _cleanup_temp() при любой ошибке
• Лимит 500MB с проверкой в процессе стриминга
• Структурированный ответ ошибки: {code, message, upload_id}

2. src/api/services/document_service.py:
• Сигнатура: file_content: bytes → temp_file_path: str
• Хеш стримингом (64KB блоки)
• os.rename() temp → uploads (атомарно)
• cleanup_stale_temp_files() — удаление файлов >30 мин
• upload_id во всех логах

3. src/models.py — добавлено поле upload_id в DocumentStatus.

Коммит: 30b0bab (428 insertions, 123 deletions).
