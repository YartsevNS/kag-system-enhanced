## 5. Что у openspg/kag лучше — и что можно взять

### 5.1. Граф знаний (CRITICAL)

**Что:** OpenSPG + Neo4j. Каждый чанк связывается с сущностями и отношениями в графе.

**Как внедрить:**
- Добавить Neo4j контейнер в docker-compose
- После чанкинга → `Extractor` (LLM) → извлечение сущностей (имена, даты, суммы, организации)
- Сохранение в Neo4j: `(Document)-[:HAS_CHUNK]->(Chunk)-[:MENTIONS]->(Entity)`
- При поиске: сначала граф (точные факты), затем векторный поиск (контекст)

**Сложность:** Средняя (3-5 дней)

### 5.2. Планировщик запросов (HIGH)

**Что:** LFPlanner — разбивает сложный вопрос на подзапросы. Пример: «Какие договоры подписал Иванов в 2024?» → [найти Иванова, найти договоры, фильтр по дате].

**Как внедрить:**
- Добавить `QueryPlanner` сервис
- Промпт: «Разбей вопрос на шаги, верни JSON: [{step, type: search|filter|aggregate}]»
- Каждый шаг → отдельный поиск → объединение результатов

**Сложность:** Средняя (2-3 дня)

### 5.3. Multi-hop рассуждение (HIGH)

**Что:** Цепочка: найти документ → найти связанные документы → сравнить. Граф позволяет «прыгать» между сущностями.

**Как внедрить:**
- После внедрения графа знаний
- `KagReasonerABC` → обход графа + LLM-рассуждение

**Сложность:** Высокая (требует графа)

### 5.4. Извлечение знаний (MEDIUM)

**Что:** LLM-экстрактор сущностей и отношений из чанков. Не просто «тип документа», а конкретные факты.

**Как внедрить:**
- Дополнить `DocumentAnalyzer` извлечением фактов
- Промпт: «Извлеки из текста: люди, организации, даты, суммы, ключевые факты. JSON.»
- Сохранять в `config_store.knowledge_units`

**Сложность:** Низкая (1 день)

### 5.5. Рефлектор качества (MEDIUM)

**Что:** После генерации ответа — проверка: нет ли галлюцинаций? Все ли факты подтверждены?

**Как внедрить:**
- `Reflector` сервис: берёт ответ + исходные чанки → проверяет соответствие
- Если низкая уверенность → переспрашивает LLM

**Сложность:** Низкая (1 день)

### 5.6. Логические формы (LOW — нишевое)

**Что:** Преобразование естественного языка в формальную логику для точных запросов (SQL-like по графу).

**Как внедрить:** Только после графа знаний. Для финансовых/юридических документов критично.

## 6. План внедрения (рекомендация)

```
Неделя 1: Извлечение знаний + Рефлектор
  ├── Дополнить DocumentAnalyzer: факты, сущности, связи
  ├── Добавить Reflector: проверка ответов
  └── Результат: точнее ответы, меньше галлюцинаций

Неделя 2: Планировщик запросов
  ├── QueryPlanner: декомпозиция сложных вопросов
  ├── Multi-step search
  └── Результат: сложные вопросы → правильные ответы

Неделя 3-4: Граф знаний (Neo4j)
  ├── Neo4j контейнер
  ├── Entity/Relation extraction
  ├── Graph-based retrieval
  └── Результат: KAG уровня openspg
```
